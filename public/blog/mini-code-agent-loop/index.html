整体视角

Mini Code 可以理解成 Claude Code 的一个最小化还原。它最值得看的不是某一个工具调用，而是围绕模型输出、工具调用、消息历史、上下文预算和 UI 状态不断协调的 Agent Loop。

这篇阅读笔记把它拆成三条主线：execute 负责推进一次模型响应，cleanup 负责一次 turn 的收尾，compact 负责上下文变长后的持续运行。

execute处理 thinking、progress、assistant 文本、tool calls、空响应和截断响应。
cleanup把消息、权限、会话、UI 和工具状态收回来，避免状态泄漏。
compact用分层策略管理上下文预算，从低成本折叠到完整 summary。

execute：一次工具调用的完整流程

模型返回的内容不一定只是 tool call。它可能带 thinking、progress、普通文本，也可能是空响应。因此 execute 的关键，是先把本轮响应拆成明确的 message 结构，再决定是否继续循环。

模型返回 tool_calls
  -> 处理 thinking / progress / assistant 文本
  -> 执行整批 tool calls
  -> 写回 assistant_tool_call 和 tool_result
  -> 如果 awaitUser，统一输出问题
  -> 否则回到下一轮 model.next(...)
thinking block保存为独立消息，但不决定流程结束。
progress 文本插入 continuation prompt，把中间状态推进成下一步动作。
普通 assistant 文本代表一次 turn 可以正常结束。
tool calls通过 ToolRegistry 校验、执行、捕获异常，并把结果结构化写回。

这里的设计让 Agent 不会因为一次空响应、截断或工具异常就立刻断掉。空响应会进入恢复提示，截断会继续请求，工具异常会变成统一的 ToolResult。

cleanup：一次 turn 结束后还要做什么

execute 完成之后，Agent 还要判断这轮到底以什么形态结束：普通 assistant 文本、awaitUser、fallback、step 上限，或者 context collapse 状态同步。

消息覆盖用 nextMessages 更新会话，让外层状态和 runAgentTurn 保持一致。
权限清除清掉 turn 级临时编辑许可，避免跨 turn 泄漏。
会话保存把新增消息、compact summary 和历史边界持久化。
UI 恢复取消忙碌状态，修正 dangling running tools。
失败兜底空响应重试耗尽或最大 step 触发时，生成明确收尾消息。
等待用户整批工具执行完后再统一抛出 awaitUser 问题。

compact：上下文变长之后如何继续运行

Mini Code 的 compact 不是一次性粗暴总结，而是按照成本和破坏性分层：能不调模型就不调模型，能不真实改 messages 就先不真实改 messages。

tool result storage超大工具输出落盘，只把路径和预览写回上下文。
microcompact清空旧工具输出，成本最低，直接改 messages。
snip compact删除安全中段历史，保护最近编辑和错误上下文。
context collapse调用模型总结旧片段，但主要改变 modelMessages 视图。
auto compact用 summary 真实替换旧历史，保留 system 和 recent tail。
manual compact用户通过 /compact、/snip、/collapse 主动介入。
system messages
context_summary
recent tail messages

这种分层策略的本质，是把上下文当成预算来管理。Agent 先做低成本、低破坏的清理，只有压力真的上来时，才进入更强的压缩。

整体体会

Mini Code 的 Agent Loop 并不神秘，但要稳定运行，需要在很多细节上持续兜底。它要推进模型继续行动，也要能在失败时恢复；要把工具结果写回上下文，也要在 turn 结束时收拾干净。

最值得学习的是两点：对异常情况的细致处理，以及对上下文预算的克制管理。一个能实际工作的 coding agent，靠的不只是 prompt 或 tool，而是一整套可恢复、可保存、可压缩的循环机制。
