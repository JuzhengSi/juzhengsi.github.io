起因

五月初，OpenAI 在 Codex 中更新了一个实验性的 Goal 功能。它允许用户设置一个目标，然后让 Codex 围绕这个目标持续工作，直到目标达成，或者明确说明为什么当前条件下无法继续推进。

这个能力本身并不陌生。今年年初，我已经在 opencode 的插件 oh-my-opencode 里接触过类似思路，它有一个叫 ralph-loop 的功能：用户给出一个目标，ralph-loop 会让模型持续尝试，直到达到目标或满足停止条件。

但 Codex Goal 的出现，仍然让我觉得很有意思。它不是简单地把“循环执行”包装成一个命令，而是把 long-term task 往更工程化的方向推了一步。换句话说，它把过去偏朴素的“让模型一直做下去”，变成了一个更接近 harness engineering 的机制：目标、状态、验证、继续执行和停止条件都被显式纳入了工作流。

Goal 到底解决什么问题

Goal 的核心思路并不复杂：用户设定一个明确的终点，Codex 根据过程中的反馈不断尝试新的方法，直到满足这个终点。

但真正难的地方在于，目标不能只是一个愿望。

如果你只说“帮我提高模型性能”，这个描述对人来说可能勉强能理解，但对一个需要持续自主工作的 coding agent 来说太宽泛了。它不知道性能指什么，不知道提升到什么程度算完成，也不知道应该用什么证据证明自己已经完成。

反过来，如果你只是想让模型改一行代码、定位一个配置项，使用 Goal 也不合适。这种场景下，普通提示就够了。强行使用 Goal，反而会让模型过度思考，浪费时间和 token。

plain prompt 请求 -> 工作 -> 结果 -> 等待

适合短任务、一次性修改、定位参数、解释错误。模型完成一轮后等待下一次输入。

goal 工作 -> 检查 -> 继续或完成

适合终点清楚但路径不确定的任务。验证结果决定继续、完成，还是诚实受阻。

它真正适合的，是那些终点清楚、路径不确定、需要多轮尝试和证据验证的任务。

如何使用 Goal

想要用好 Goal，关键是两件事：使用时机和目标设计。

什么时候适合使用

01性能优化，例如把 p95 延迟降低到某个阈值以下。
02修复偶发失败的测试，并确认失败原因或修复证据。
03迁移一组复杂依赖，同时保持现有测试通过。
04做基准驱动的调优，每一轮都要根据指标决定下一步。

这些任务都有一个共同点：最终状态可以定义，但中间路径不确定。Codex 需要探索、试错、运行验证，再决定是否继续。

一个好的 Goal 应该包含什么

结果工作完成时什么应该为真。
验证面用测试、基准、报告、产物、命令输出或源材料证明完成。
约束执行过程中哪些行为、质量或兼容性不能回退。
边界Codex 可以使用哪些文件、工具、数据、仓库或资源。
迭代策略每次尝试之后，Codex 应该如何决定下一步。
受阻停止条件没有可靠路径时，Codex 应何时停止并报告什么。

这六个要素不一定每次都要写满，但至少应该有“结果”和“验证面”。没有结果，模型不知道什么叫完成；没有验证面，模型很容易把“看起来完成了”误判成真正完成。

Goal 模板

一个比较薄的 Goal 可以这样写：

/goal 优化这个系统，将 p95 延迟降到 120 ms 以下，同时不让正确性测试回退。

它已经包含了结果和约束，但仍然不够强。更完整的版本可以写成：

/goal 将 p95 结账延迟降到 120 ms 以下，由 checkout 基准测试验证，同时保持正确性套件为绿色。只使用 checkout 服务、基准 fixture 和相关测试。在每次迭代之间，记录改了什么、基准测试显示了什么，以及下一个最值得尝试的实验。如果基准测试无法运行或没有有效路径，停止并报告已尝试的路径、收集到的证据、阻塞点和下一步需要的输入。

这个版本就更像一个可执行契约。它不仅说明要做什么，还说明怎么验证、不能破坏什么、允许修改哪里、每一轮如何继续，以及什么时候应该承认受阻。

以后设计 Goal 时，可以把它当作一份任务契约，而不只是一段更长的 prompt。
/goal <期望结束状态>，由 <具体证据> 验证，同时保持 <约束>。使用 <允许的输入、工具或边界>。在迭代之间，<Codex 应如何选择下一个最佳动作>。如果受阻或没有有效路径，<Codex 应报告什么，以及什么可以解锁进展>。

让 Codex 帮你设计 Goal

除了手动设计 Goal，也可以先让 Codex 帮你把自然语言需求转成 Goal 草稿。

第一步用自然语言描述你想做的事情，并让 Codex 把它转换成目标草稿。
第二步审查这个草稿，在真正激活 Goal 之前，收紧成功条件、验证面、约束和受阻停止条件。

例如可以这样问：

帮我把这件事转成一个强 `/goal`：我希望 Codex 持续处理这个偶发失败的 checkout 测试，直到我们要么用证据修复它，要么能清楚解释是什么阻碍了进展。

这个步骤非常重要。很多时候，用户一开始说出来的是“愿望”，不是“目标”。Codex 可以帮助我们把愿望拆成可执行、可验证、可停止的契约。

从 ralph-loop 到 Codex Goal

我觉得 Codex Goal 最值得关注的地方，不在于它能“循环执行”。循环本身并不新鲜，ralph-loop 已经展示过类似能力。

真正的变化在于，Codex Goal 把这个循环放进了更完整的运行时机制里。它不只是提示模型继续，而是让目标成为线程中的持久状态。模型后续的工作、验证、完成判断和受阻报告，都围绕这个状态展开。

ralph-loop 更像一个插件层面的 continuation loop：它告诉模型，只要没有输出完成标记，就继续工作。Codex Goal 则更像一个被运行时承认的任务契约：它有目标文本，有状态，有预算，有继续条件，也有完成和受阻的生命周期。

ralph-loop 插件层面的继续循环

更朴素，重点是让模型在未完成时继续执行。

Codex Goal 运行时承认的任务契约

有状态、预算、继续条件，以及完成和受阻的生命周期。

对 autoresearch 的启发

今年年初，Andrej Karpathy 在 GitHub 上开源了一个名为 autoresearch 的项目。简单来说，它希望让大模型围绕一个研究目标持续尝试，在给定指标上不断改进。

这个项目里的流程很有代表性：初始化、定义目标、迭代研究、评估、记录过程和错误，最后总结报告。

这其实就是一个典型的 long-term task 流程。但当时我看到它时，一个很直接的感受是：流程是完整的，但还缺少一个稳定驱动流程执行的执行者。

Codex Goal 正好补上了这个位置。如果把 autoresearch 的研究流程和 Codex Goal 结合起来，Goal 可以承担“持续推进”的角色。它让模型不是完成一次回答，而是在指标、证据和停止条件的约束下持续工作。

Long-term task 仍然远未解决

无论是 ralph-loop 还是 Codex Goal，它们都说明 long-term task 正在变成 coding agent 的一个重要方向。这个方向的核心问题是：如何让大模型可靠地完成一个需要数小时甚至数十小时的任务。

memory模型如何保留关键经验，而不是被上下文窗口限制。
state系统如何记录当前进度、失败路径、已验证事实和剩余风险。
planning模型如何把大目标拆成阶段性策略。
check系统如何判断一次尝试是否真的有效。
recovery当任务受阻时，模型如何诚实停止，而不是假装完成。
management多个 Goal 之间如何描述依赖、优先级和执行顺序。

Codex Goal 是这个方向上的一个关键进展，但距离真正的 aha moment 还有很长的路。

还缺什么

我在日常使用 Goal 时，也遇到了一些明显的痛点。第一个问题是，目前一次通常只支持一个 Goal。如果我有多个存在依赖关系的目标，就只能等前一个完成后，再手动切换到下一个。这背后缺少一个 Goal 管理系统，用来描述多个 Goal 之间的依赖、优先级和执行顺序。

第二个问题是，Goal 的设计质量会直接影响执行质量。Goal 写得不好，模型就会在错误的边界里努力，最后可能事倍功半。

为了解决这个问题，我自己设计了一个 draft-goal skill。它依赖官方文档里对强 Goal 的定义，通过用户最初的自然语言描述，以及后续必要的澄清问题，帮助用户生成一个更完整的 Goal 提示词。

未来使用 coding agent 时，我们不只是要会写 prompt，也要会写 goal。Prompt 更像一次请求，而 Goal 更像一份任务契约。对于长时任务来说，后者的重要性会越来越高。

结语

从 ralph-loop 到 Codex Goal，我看到的是 coding agent 从“能连续工作”走向“能被工程化管理”的过程。

过去我们更关心模型一次回答得好不好。现在，一个更重要的问题开始浮现：模型能不能在长期任务中持续积累证据、修正方向、管理状态，并在合适的时候诚实地说完成或受阻。

Codex Goal 还不是终点，但它给出了一个很有价值的方向：不要只让模型继续做，而是给它一个可验证、可审计、可停止的目标。
